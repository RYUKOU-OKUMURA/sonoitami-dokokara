# 技術スタック整理

## 1. 技術選定の前提

このプロジェクトのMVPは、派手な3D表現ではなく、シナリオ分岐と見立てロジックが主役です。したがって、技術選定では以下を優先します。

- 分岐ロジックを安全に実装できる
- シナリオ追加がしやすい
- テキスト中心でもリッチに見せられる
- 将来のAI拡張に接続しやすい
- 初期コストを上げすぎない

## 2. 結論

MVPの推奨構成は以下です。

- フロントエンド基盤: `Next.js + React + TypeScript`
- スタイリング: `Tailwind CSS`
- 状態管理: `XState`
- コンテンツ定義: `JSON` もしくは `TypeScriptオブジェクト + Zodバリデーション`
- テスト: `Vitest` と `Playwright`
- 保存: `localStorage`
- デプロイ: `Vercel`

## 3. この構成を推す理由

### Next.js + React + TypeScript

理由:

- MVPではクライアント中心のアプリで十分だが、将来APIやAI機能を追加しやすい
- ルーティング、ビルド、配信の基盤を最初から安定させやすい
- Reactのコンポーネント設計で、画面とロジックを分離しやすい
- TypeScriptにより、シナリオデータや分岐条件の破綻を早めに検出できる

補足:

- いま最速だけを優先するなら `Vite + React` も候補になる
- ただし本件は将来的な拡張が見えているため、最初から `Next.js` に寄せた方が手戻りが少ない

### Tailwind CSS

理由:

- テキスト中心のUIでも密度や余白を調整しやすい
- MVP段階でデザイン速度を落としにくい
- フェーズ表示、選択肢カード、スコア表示などを素早く作れる

補足:

- UIライブラリは最初から増やしすぎない
- 独自の世界観を出したいので、MVPでは汎用コンポーネント依存を最小限に抑える

### XState

理由:

- 本作は「問診 -> 仮説 -> 評価 -> 施術 -> 結果」という状態遷移が明確
- 分岐や行動回数制限を状態機械で表現すると、実装と仕様の対応が明快になる
- 後でシナリオが増えても、フェーズ遷移の整合性を保ちやすい

補足:

- 単純なグローバル状態共有だけなら `Zustand` でも足りる
- ただし今回は、単なる値保持より「進行ルールの管理」が重要なので `XState` の方が適している

### JSON または TypeScriptオブジェクト + Zod

理由:

- シナリオをデータ主導で管理したい
- 選択肢の抜けや誤記を、実行前に検証したい
- 将来CMSやAI生成データと接続する際も移行しやすい

推奨:

- MVPでは、編集しやすさと型安全性のバランスから `TypeScriptオブジェクト + Zod` を第一候補にする
- データ担当者がコードを書かない体制になった時点で、`JSON` または `YAML` への分離を検討する

### Vitest と Playwright

理由:

- スコア計算や分岐判定は単体テストしやすい
- プレイ導線が最後まで通るかはE2Eで確認したい
- 本作はロジック破綻が体験品質に直結するため、UI以上にゲームルールを守るテストが重要

## 4. アーキテクチャ方針

MVPでは、UIとゲームロジックとシナリオデータを明確に分離する。

### 推奨レイヤー

- `app/`
  - ルーティングとページ構成
- `src/features/game/`
  - フェーズ進行、スコア計算、結果判定
- `src/features/scenario/`
  - シナリオ読み込みとデータ変換
- `src/components/`
  - UIコンポーネント
- `src/content/scenarios/`
  - シナリオデータ
- `src/lib/`
  - 共通ユーティリティ

### 重要な分離

- シナリオ文言をUIコンポーネントに埋め込まない
- スコア計算を画面イベントの中に散らさない
- 結果判定を1か所に集約する

## 5. 推奨データモデル

最低限、以下の構造を持てるようにする。

- scenario
- patient
- interviewChoices
- hypothesisChoices
- evaluationChoices
- treatmentChoices
- scoreRules
- endings
- feedback

この構造にしておくと、王道シナリオ以外も同じ土台で追加できる。

## 6. 保存方針

MVPは `localStorage` で十分です。

保存対象:

- 現在のフェーズ
- 選択履歴
- スコア
- クリア状況

まだ不要なもの:

- アカウント紐付け
- サーバー保存
- 複数端末同期

## 7. デザイン実装方針

- ノベルゲーム風の読みやすさを優先する
- 患者情報、会話、選択肢、スコアを同一画面で把握できる構成にする
- 身体図や部位タグは補助情報として使う
- 医療機器UIのような硬さではなく、物語と観察の中間のトーンにする

## 8. 現時点で採らない技術

### Unity / Godot

不採用理由:

- MVPに対して重い
- 分岐と文脈設計より、演出側の実装コストが先に立ちやすい

### Supabase

MVPでは不採用理由:

- 保存要件がまだ軽い
- 認証やDB設計が先に来ると、ゲーム体験の検証が遅れる

将来の利用可能性:

- ユーザー進行保存
- シナリオ配信管理
- AI講評履歴保存

### LLMリアルタイム対話

MVPでは不採用理由:

- 品質制御が難しい
- ゲーム体験の基礎設計が固まる前に入れると、面白さの原因がぼやける

## 9. 技術的リスクと対策

### リスク: 分岐の複雑化

対策:

- シナリオデータを型とスキーマで検証する
- 遷移制御を状態機械で一元管理する

### リスク: 文言修正でロジックが壊れる

対策:

- 文言とIDを分離する
- 条件分岐はIDベースで判定する

### リスク: 追加シナリオで構造が破綻する

対策:

- MVPから「シナリオを差し替えれば遊べる」設計に寄せる
- シナリオ固有処理を増やしすぎない

## 10. 実装開始前に固めるべき事項

以下が確定したら、実装を始めやすい。

1. MVPの王道シナリオ詳細
2. フェーズごとの選択回数制限
3. スコアの計算ルール
4. エンディング分岐条件
5. 画面ワイヤー

## 11. 推奨する次の一手

次はコードを書く前に、以下の順で詰めるのが妥当です。

1. 王道シナリオの症例設定を1本分、完全に文章化する
2. 問診、評価、施術の選択肢を洗い出す
3. 各選択肢がどのスコアにどう影響するかを表にする
4. その表をもとに、画面ワイヤーを作る

# 技術スタック整理

## 1. 技術選定の前提

このプロジェクトのMVPは、正式版ではなく `限定公開MVP` です。派手な演出や周辺機能よりも、王道シナリオ1本を最後まで遊べることと、UIとゲームループの手応えを素早く検証できることを優先します。

したがって、技術選定では以下を優先します。

- 1画面中心のプレイ体験を短期間で形にできる
- 分岐ロジックとスコア判定を安全に実装できる
- 王道シナリオ1本で成立した後に、推理系や好感度重視系へ拡張しやすい
- 保存やAIが未実装でもMVPとして成立する
- 初期コストを上げすぎず、`build-ready` 後すぐ実装に入れる

## 2. 結論

現時点の推奨構成は以下です。

- フロントエンド基盤: `Next.js + React + TypeScript`
- スタイリング: `Tailwind CSS`
- 状態管理: `React state / useReducer`
- コンテンツ定義: `TypeScriptオブジェクト + Zodバリデーション`
- テスト: `Vitest` と最小限の `Playwright`
- デプロイ: `Vercel`

MVPでは見送り、必要になった時点で追加する候補:

- 保存: `localStorage` は `Nice to have / v0.2候補`
- 状態機械: `XState` は複雑化時の昇格先
- バックエンド保存: `Supabase` などは後続検討

## 3. この構成を推す理由

### Next.js + React + TypeScript

理由:

- 1画面中心のWebアプリでも、ページ基盤と配信方法を最初から安定させやすい
- ReactでUIとゲームロジックを分離しやすい
- TypeScriptでシナリオデータ、分岐条件、スコア計算の破綻を早めに検出しやすい
- 限定公開MVPの後、そのまま機能追加して育てやすい

補足:

- 最速だけを優先するなら `Vite + React` も候補になる
- ただし今回は、限定公開後も継続して育てる前提があるため、文書上の第一候補は `Next.js` のままにする

### Tailwind CSS

理由:

- テキスト中心のUIでも密度や余白を素早く調整しやすい
- フェーズ表示、選択肢カード、スコア表示などを短期間で組みやすい
- 1画面構成の情報密度を試しながら調整しやすい

補足:

- MVPではUIライブラリ依存を増やしすぎない
- 独自の世界観を出すため、汎用コンポーネントの導入は最小限に留める

### React state / useReducer

理由:

- MVPの範囲は `シナリオ1本` `1画面構成` `固定フェーズ進行` なので、まずは軽い状態管理で十分
- `currentPhase` `selectedChoices` `discoveredInfo` `scores` `derivedEnding` を素直に扱いやすい
- 初回プレイアブルを作る段階で、状態管理自体の抽象度を上げすぎずに済む

補足:

- 分岐や条件が増え、遷移の可視化が必要になったら `XState` を導入候補にする
- MVPの時点では、状態機械を必須前提にしない

### TypeScriptオブジェクト + Zod

理由:

- シナリオをデータ主導で管理したい
- `choice ID`、`flag`、`score rule`、`ending condition` を文言から分離して検証したい
- 将来の推理系や好感度重視系で、スコア軸や講評の差し替えを扱いやすい

推奨:

- MVPでは `TypeScriptオブジェクト + Zod` を第一候補にする
- データ担当者がコードを書かない運用に変わった時点で、`JSON` や `YAML` への分離を検討する

### Vitest と最小限の Playwright

理由:

- スコア計算、結果分岐、選択肢の影響は `Vitest` で主に検証しやすい
- `Playwright` は最低1本の通し導線と代表プレイパターンのスモーク確認に絞ると、MVPに対して過剰になりにくい
- 本作はUIの見た目以上に、分岐ロジックの破綻が体験品質に直結する

### Vercel

理由:

- Next.js と相性がよく、限定公開MVPの公開先として扱いやすい
- インフラ構築を重くせず、プレイアブル検証に集中しやすい

補足:

- `Vercel` は推奨デプロイ先だが、`build-ready` 条件には含めない

## 4. アーキテクチャ方針

MVPでは、UI、ゲームロジック、シナリオデータを明確に分離する。

### 推奨レイヤー

- `app/`
  - ルーティングとページ構成
- `src/features/game/`
  - フェーズ進行、選択反映、スコア計算、結果判定
- `src/features/scenario/`
  - シナリオ読み込み、データ変換、バリデーション
- `src/components/`
  - プレイUI、選択肢、スコア表示などのUIコンポーネント
- `src/content/scenarios/`
  - シナリオデータ
- `src/lib/`
  - 共通ユーティリティ

### 重要な分離

- シナリオ文言をUIコンポーネントに埋め込まない
- スコア計算をイベントハンドラの中に散らさない
- 結果判定を1か所に集約する
- スコア軸の意味や表示ラベルを、シナリオ差し替えで変更可能にする

### MVPで持つ最小状態

- `currentPhase`
- `selectedChoices`
- `discoveredInfo`
- `scores`
- `derivedEnding`

## 5. 推奨データ契約

最低限、以下の構造を持てるようにする。

- `scenario`
- `patient`
- `choices`
- `flags`
- `scoreRules`
- `endingConditions`
- `feedback`
- `scoreAxisDefinitions`

データ設計では以下を守る。

- 表示文言と判定条件を分離する
- 各選択肢は表示文言とは別に `choice ID` を持つ
- 分岐判定は `ID` や `flag` を基準に行う
- `scoreRules`、`endingConditions`、`feedback` は独立して管理する
- 王道シナリオ以外でも、スコア軸の意味や表示ラベルを差し替えられるようにする

この構造にしておくと、王道シナリオの後に推理系や好感度重視系を追加しやすい。

## 6. 保存方針

限定公開MVPでは、保存機能は必須ではない。まずは保存なしでも1プレイが成立することを優先する。

`localStorage` は次の条件を満たした時点で追加候補とする。

- 限定公開で途中離脱や再開ニーズが確認された
- プレイ時間や分岐の複雑さが増えた
- リロード復元が体験価値に直結すると判断できた

`localStorage` を追加する場合の保存対象:

- 現在のフェーズ
- 選択履歴
- 取得情報
- スコア
- クリア状況

まだ不要なもの:

- アカウント紐付け
- サーバー保存
- 複数端末同期

## 7. デザイン実装方針

- ノベルゲーム風の読みやすさを優先する
- 患者情報、会話、選択肢、取得情報、スコアを同一画面で把握できる構成にする
- 初回プレイアブルでは、情報過多よりも理解しやすさを優先する
- 医療機器UIのような硬さではなく、物語と観察の中間のトーンにする

## 8. MVPでは見送るが候補に残す技術

### XState

MVPで第一候補にしない理由:

- シナリオ1本、1画面、固定フェーズ進行の段階では抽象度が高すぎる可能性がある
- まずは `state / useReducer` で体験とロジックを固める方が速い

導入を再検討する条件:

- シナリオ数が増える
- 条件分岐や戻り遷移が増える
- 状態遷移を図として可視化したくなる

### Supabase

MVPで見送る理由:

- 保存要件が必須ではない
- 認証やDB設計が先に来ると、プレイ体験の検証が遅れる

将来の利用可能性:

- ユーザー進行保存
- シナリオ配信管理
- AI講評履歴保存

### LLMリアルタイム対話

MVPで見送る理由:

- 品質制御が難しい
- ゲーム体験の核が固まる前に入れると、面白さの原因がぼやける

### Unity / Godot

MVPで見送る理由:

- 1画面中心のWebシミュレーションに対して重い
- 演出実装のコストが、分岐や文脈設計より先に立ちやすい

## 9. 技術的リスクと対策

### リスク: 分岐の複雑化

対策:

- シナリオデータを型とスキーマで検証する
- 判定ロジックを1か所に集約する
- 初期バランス案と固定仕様を文書上で分離する

### リスク: 文言修正でロジックが壊れる

対策:

- 文言と `choice ID` を分離する
- 条件分岐は `ID` と `flag` ベースで判定する

### リスク: シナリオ追加で構造が破綻する

対策:

- MVPから「シナリオを差し替えれば遊べる」設計に寄せる
- スコア軸の意味や表示ラベルを差し替え可能にする
- 王道シナリオ固有の処理を増やしすぎない

## 10. 実装開始条件

この技術スタック文書だけでは、まだ実装着手可能とはみなさない。以下の4点が揃った時点を `build-ready` とする。

1. 王道シナリオの全文テキスト
2. 問診、仮説、評価、施術、説明方針の具体内容と、各選択肢のID、フラグ、スコア影響表
3. 4エンディングの名称と講評文
4. 1画面プレイUIのワイヤーフレーム

補足:

- フェーズごとの選択回数制限や結果閾値は、固定仕様ではなく初期バランス案として扱う

## 11. 推奨する次の一手

次はコードを書く前に、以下の順で詰めるのが妥当です。

1. 王道シナリオの全文を固める
2. 各選択肢の `ID`、`flag`、`score rule`、`ending condition` を表にする
3. 4エンディングの講評文を固める
4. 1画面プレイUIのワイヤーを作る
5. その後に実装へ入る
